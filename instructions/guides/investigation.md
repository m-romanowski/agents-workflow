# Investigation

Use this guide for debugging, audits, code review, diagnosis, discovery, and repository analysis.

## Expectations
- Start with findings, questions, and options.
- Prefer analysis before implementation.
- Do not convert investigation into code changes without explicit follow-up approval.
- If the operator later approves implementation, also load the relevant implementation guides, and create or update an active task directory in `instructions/work/` before editing the codebase.
- Route `wf:review` here by default unless a more specific review guide exists.
- If the investigation concerns a hypothetical future feature, load `feature-discovery.md` only when the operator wants option shaping beyond evidence gathering.
- When investigation and feature discovery both apply, treat investigation as evidence-gathering and feature discovery as decision-shaping, but keep one guide primary to limit hot-loaded context.

## Output Shape
- Current understanding
- Evidence or observations
- Risks or uncertainties
- Options for next steps
